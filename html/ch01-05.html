Part 1: What Is a System?

Chapter 1: The City Analogy

The Analogy

Imagine standing at the top of a tall building, looking down at a city.

You can see roads. Cars move along them, carrying people and goods from one place to another. You can see buildings: offices, homes, factories, warehouses, each with a purpose. There are water pipes underground, invisible, quietly delivering water to every tap. There are electrical cables forming a power grid, keeping every light on. And there is a postal system: trucks collecting letters, sorting offices routing them, postmen delivering them to the right door.

Here is the central insight of this book: a software system is exactly like a city.

Every piece of technology you have ever used, whether a website, an app, a game, or a smart speaker, is a city. It has roads, buildings, water pipes, a power grid, and a postal system. The names are different, but the underlying ideas are identical.

Before you ever touch a computer, you already understand how systems work, because you live inside one every day.

The Concept

A system is a collection of parts working together toward a shared goal. A city's goal is to let people live, work, and move efficiently. A software system's goal might be to let you send a message to a friend, or to allow an AI agent to find the right tool for a task.

Every system, regardless of complexity, can be broken down into the same five components:

RoadsNetworks (the internet)Moves information from one place to another
BuildingsServersStores data and performs work
Post officesAPIsHandles requests and sends responses
Filing cabinetsDatabasesRemembers things permanently
Power gridCloud infrastructureKeeps everything running

These five components appear throughout every chapter in this book. Study them carefully, because every system you will ever build or analyse reduces to some combination of these five things.

The Two Systems in This Book

Throughout this book, we will study a system made up of two parts working together.

The first part is the Tool Registry: a backend server written in Go that maintains a searchable catalogue of AI tools. It stores information about each tool, who made it, and how to use it. Any AI agent can query the registry to discover what tools are available.

The second part is the AI assistant: a program written in Python and Rust that reads the registry, selects the right tools, and uses them to complete tasks on your behalf.

Here is how the two parts connect:

You (the human)
    ↓
AI Assistant (selects and uses tools)
    ↓  queries: "what tools are available?"
Tool Registry (maintains the catalogue)
    ↓  returns a list of registered tools
AI Assistant picks a tool and uses it
    ↓
The tool acts in the real world
   (reads a file, queries a database, calls another service)

This chain, from you through the AI assistant to the registry and out to the world, is the thread that runs through every chapter. By the end of this book, you will understand every component in it.

"""
system_sim.py
Simulates the five components of a software system in a single script.
Each component is a Python function that logs what it does.
"""

import json
import time

# ── COMPONENT 1: DATABASE (the filing cabinet) ────────────────
DATABASE = {
    "tools": [
        {"id": "file-reader",  "name": "File Reader",  "version": "1.0"},
        {"id": "web-search",   "name": "Web Search",   "version": "2.1"},
        {"id": "calculator",   "name": "Calculator",   "version": "1.3"},
    ]
}

def database_query(query_type, params=None):
    """Retrieve records from the in-memory database."""
    print(f"  [DATABASE]  Received query: {query_type}")
    if query_type == "list_tools":
        results = DATABASE["tools"]
        print(f"  [DATABASE]  Returning {len(results)} records.")
        return results
    if query_type == "get_tool" and params:
        match = [t for t in DATABASE["tools"] if t["id"] == params.get("id")]
        return match[0] if match else None
    return None


# ── COMPONENT 2: SERVER (the building) ───────────────────────
def server_handle(request):
    """Process an incoming request and return a response."""
    print(f"  [SERVER]    Processing request: {request['path']}")
    if request["path"] == "/tools":
        data = database_query("list_tools")
        return {"status": 200, "body": {"tools": data, "total": len(data)}}
    if request["path"].startswith("/tools/"):
        tool_id = request["path"].split("/")[-1]
        data = database_query("get_tool", {"id": tool_id})
        if data:
            return {"status": 200, "body": data}
        return {"status": 404, "body": {"error": "Tool not found"}}
    return {"status": 404, "body": {"error": "Unknown path"}}


# ── COMPONENT 3: API (the post office) ───────────────────────
def api_receive(method, path, body=None):
    """
    The API is the front door of the server.
    It receives requests and passes them to the server.
    """
    print(f"  [API]       {method} {path}")
    request = {"method": method, "path": path, "body": body}
    response = server_handle(request)
    print(f"  [API]       Responding with status {response['status']}")
    return response


# ── COMPONENT 4: NETWORK (the road) ──────────────────────────
def network_send(destination, method, path, body=None):
    """
    The network carries requests from the client to the API.
    In real life this is the internet. Here it is a function call.
    """
    print(f"  [NETWORK]   Sending {method} {path} to {destination}")
    time.sleep(0.05)   # simulate a small network delay
    response = api_receive(method, path, body)
    print(f"  [NETWORK]   Response delivered back to client.")
    return response


# ── COMPONENT 5: CLIENT (you) ─────────────────────────────────
def client_request(path):
    """The client is the starting point: it initiates every request."""
    print(f"\n[CLIENT]  Requesting: {path}")
    response = network_send("tool-registry.local", "GET", path)
    print(f"[CLIENT]  Received: {json.dumps(response['body'], indent=2)}")
    return response


# ── RUN THE SIMULATION ────────────────────────────────────────
print("=" * 58)
print("SYSTEM SIMULATION: Five Components in Action")
print("=" * 58)

print("\n--- Request 1: List all tools ---")
client_request("/tools")

print("\n--- Request 2: Get details for one tool ---")
client_request("/tools/web-search")

print("\n--- Request 3: Request something that does not exist ---")
client_request("/tools/does-not-exist")

print("\n" + "=" * 58)
print("Every request above passed through all five components:")
print("  CLIENT → NETWORK → API → SERVER → DATABASE → back")
print("=" * 58)

Chapter 2: Clients and Servers

The Analogy

You walk into a restaurant and sit down. A waiter hands you a menu. You decide what you want, tell the waiter, and wait. The waiter walks to the kitchen, passes your order to the chef, and returns with your meal.

Consider what happened in each step:

  • You never entered the kitchen. You never saw the chef. You never touched the stove.
  • The waiter acted as the intermediary, receiving your request and delivering the result.
  • The kitchen performed the real work: storing ingredients, cooking the meal, and preparing the plate.

This is precisely how the internet works, every single time.

  • You are the client: the one who makes requests.
  • The waiter is the API: the agreed channel between you and the kitchen.
  • The kitchen is the server: the one who does the work and holds the data.

The Concept

In any software system, every interaction involves two roles:

The Client asks for things. Your phone is a client. Your web browser is a client. An AI assistant is a client when it requests tools from a registry. Clients do not store the primary data; they request it from servers.

The Server answers requests and holds the data. A tool registry is a server. A video platform's computers are servers. Your school's grading system runs on a server. Servers wait silently until a client contacts them.

This relationship is called the client-server model, and it is the foundation of almost everything on the internet.

The client and the server never need to be in the same room, or even the same country. They communicate over the internet using agreed-upon languages called protocols, which Chapter 3 explains.

One Server, Many Clients

A server can respond to thousands of clients at the same time. Right now, millions of people are watching videos on a platform like YouTube. Every one of them has a client sending requests to the same servers. Those servers handle all the requests simultaneously, like a kitchen with fifty chefs each working on a different order.

A Component Can Be Both

A single piece of software can be a client and a server at the same time, depending on which direction you are looking. Consider the chain from Chapter 1:

You (client)
    → AI Assistant
        → Tool Registry (server)

From your perspective, the AI assistant is the server; it responds to your instructions. From the AI assistant's perspective, the Tool Registry is the server; the assistant sends queries to it. The AI assistant is simultaneously a server to you and a client to the registry.

This is not contradictory. It is simply a sign that real systems have layers. Each layer is a client to everything above it and a server to everything below.

The Real Code

The following is a simplified version of what a server's startup code looks like in Go. When this code runs, the server opens for business and begins accepting requests from clients.

// tool-registry/cmd/server/main.go
// This is the entry point: the moment the server opens for business.
func main() {
    // Load configuration (which port to listen on, which database to use)
    cfg := config.Load()

    // Connect to the database (where all tool records are stored)
    db := database.Connect(cfg.DatabaseURL)

    // Begin accepting requests from clients
    server := api.NewServer(cfg, db)
    server.Start(cfg.Port)
}

Each line corresponds to a step in opening the restaurant:

  1. Read the configuration: what address to listen on, what database to use.
  2. Connect to the database: where all tool records are stored.
  3. Start accepting requests from clients.

Once this code runs, any client, including an AI assistant, can send requests to this server.

"""
server.py  -  A minimal HTTP server representing the Tool Registry.
Run this in one PowerShell window, then use client.py in a second window.
"""
from http.server import HTTPServer, BaseHTTPRequestHandler
import json

TOOLS = [
    {"id": "file-reader",  "name": "File Reader",  "description": "Reads files from disk"},
    {"id": "web-search",   "name": "Web Search",   "description": "Searches the internet"},
    {"id": "calculator",   "name": "Calculator",   "description": "Evaluates expressions"},
]

class RegistryHandler(BaseHTTPRequestHandler):
    def log_message(self, format, *args):
        print(f"  [SERVER] {format % args}")

    def send_json(self, status, data):
        body = json.dumps(data, indent=2).encode()
        self.send_response(status)
        self.send_header("Content-Type", "application/json")
        self.send_header("Content-Length", len(body))
        self.end_headers()
        self.wfile.write(body)

    def do_GET(self):
        if self.path == "/tools":
            self.send_json(200, {"tools": TOOLS, "total": len(TOOLS)})
        elif self.path.startswith("/tools/"):
            tool_id = self.path.split("/")[-1]
            match = next((t for t in TOOLS if t["id"] == tool_id), None)
            if match:
                self.send_json(200, match)
            else:
                self.send_json(404, {"error": f"No tool with id '{tool_id}'"})
        elif self.path == "/healthz":
            self.send_json(200, {"status": "ok"})
        else:
            self.send_json(404, {"error": "Path not found"})

print("Tool Registry server running at http://localhost:8800")
print("Waiting for client requests. Press Ctrl+C to stop.\n")
HTTPServer(("localhost", 8800), RegistryHandler).serve_forever()

"""
client.py  -  Demonstrates the client side of the client-server model.
The server (server.py) must be running in another window first.

import urllib.request
import json

BASE = "http://localhost:8800"

def request(path):
    """Send a GET request to the local server and return the parsed response."""
    url = f"{BASE}{path}"
    try:
        with urllib.request.urlopen(url, timeout=5) as resp:
            return resp.status, json.loads(resp.read())
    except urllib.error.HTTPError as e:
        return e.code, {"error": str(e.reason)}

Chapter 3: Protocols: The Language of Machines

The Analogy

Imagine two people trying to build a house together, one speaking only Japanese, the other speaking only Portuguese. They stand in front of a pile of bricks and tools and nothing happens. Neither can understand the other's instructions.

They make a decision: both will learn to communicate in English. Not because English is the best language, but because both have agreed to use it. Now when one says "hand me the hammer," the other understands. Work begins.

That agreement, "we will use English," is a protocol. A protocol is not about what to say; it is about how to say it so that both sides can understand.

Machines face exactly this problem. A phone in your pocket, a server in Iceland, and a thermostat on your wall are built by different companies using different hardware running different software. They need an agreed-upon language to communicate. That language is a protocol.

The Concept

A protocol is a set of rules that two systems agree to follow when communicating. Protocols define three things:

  • The format messages must take (comparable to agreeing to write letters rather than speak aloud)
  • The order in which things happen (comparable to agreeing that you introduce yourself before conducting business)
  • How to handle errors (comparable to agreeing on what "I did not understand" looks like)

The Protocols That Power the Web

HTTP (HyperText Transfer Protocol) is the language of the web. When you type a web address into a browser, your browser sends an HTTP request to a server. The server sends back an HTTP response. Every webpage you have ever loaded was delivered using HTTP.

JSON (JavaScript Object Notation) is not a protocol but a format, a structured way of representing data so that any machine can read it:

{
  "name": "File Reader",
  "version": "1.2.0",
  "description": "Reads files from your computer"
}

JSON is human-readable and machine-readable simultaneously. You can look at it and understand it without any special software.

MCP (Model Context Protocol) is the central protocol in this book. MCP was designed specifically to allow AI agents to communicate with tools. It defines how an agent announces that it wants to use a tool, how the tool responds, and how errors are reported. Without MCP, every AI agent would need to learn a different language for each tool it wanted to use. With MCP, one protocol works for all tools.

System Diagram

PROTOCOL = A shared agreement on message format

WITHOUT a protocol:
  Client: "Hello give me the tools"     Server: ???  (does not understand)
  Client: "Tools please list them all"  Server: ???  (still confused)
  RESULT: nothing works

WITH HTTP + JSON:
  Client: GET /tools HTTP/1.1           Server: 200 OK
          Accept: application/json             {"tools": [...]}
  RESULT: both sides understand, every time

Protocol stack used in this book:
  ┌──────────────────────────────────┐
  │  MCP Protocol                   │  ← AI agents use this to call tools
  ├──────────────────────────────────┤
  │  HTTP / HTTPS                   │  ← How requests travel
  ├──────────────────────────────────┤
  │  JSON                           │  ← How data is formatted
  ├──────────────────────────────────┤
  │  TCP/IP                         │  ← How packets are delivered reliably
  └──────────────────────────────────┘

Each layer depends on the one below it.
TCP/IP carries the bytes. JSON structures them.
HTTP defines the request/response format. MCP defines the tool conversation.

The Real Code

The Tool Registry stores information about tools in a format defined by a JSON schema. A schema is a set of rules that specifies exactly what a valid tool entry must look like. The following is a simplified version:

// tool-registry/internal/validators/schemas/v1.json
{
  "required": ["name", "description", "packages"],
  "properties": {
    "name": {
      "type": "string",
      "description": "Human-readable name of the tool"
    },
    "description": {
      "type": "string",
      "description": "What this tool does"
    },
    "packages": {
      "type": "array",
      "description": "How to install this tool (npm, pip, docker, etc.)"
    }
  }
}

This schema is a contract. If you try to register a tool without a name, the registry rejects it because you have broken the protocol. You have attempted to communicate in a way the other side cannot understand.

"""
validate.py
Checks whether a tool manifest follows the protocol rules.
This is the same kind of validation the Tool Registry performs
on every submission before accepting it.
"""
import json
import re

def validate(path):
    print(f"Validating: {path}\n")
    errors = []

    with open(path) as f:
        data = json.load(f)

    # Rule 1: Required fields must be present
    required_fields = ["name", "description", "version", "packages"]
    for field in required_fields:
        if field not in data:
            errors.append(f"Missing required field: '{field}'")
        else:
            print(f"  PASS  '{field}' is present")

    # Rule 2: Name must follow the format namespace/tool-name
    name = data.get("name", "")
    pattern = re.compile(r'^[a-zA-Z0-9][a-zA-Z0-9._-]*/[a-zA-Z0-9][a-zA-Z0-9._-]*$')
    if not pattern.match(name):
        errors.append(f"Invalid name format: '{name}': use 'namespace/tool-name'")
    else:
        print(f"  PASS  name format is valid: '{name}'")

    # Rule 3: Version must be an exact number, not a range
    version = data.get("version", "")
    if re.match(r'^[\^~>=<]', version):
        errors.append(f"Invalid version: '{version}': must be exact, e.g. '1.2.3'")
    elif not re.match(r'^\d+\.\d+\.\d+', version):
        errors.append(f"Invalid version: '{version}': must follow semver, e.g. '0.1.0'")
    else:
        print(f"  PASS  version is valid: '{version}'")

    # Rule 4: Packages must be a non-empty list
    packages = data.get("packages", [])
    if not isinstance(packages, list) or len(packages) == 0:
        errors.append("Invalid packages: must be a non-empty list")
    else:
        print(f"  PASS  packages: {len(packages)} package(s) defined")

    # Rule 5: Description must be meaningful (at least 20 characters)
    desc = data.get("description", "")
    if len(desc) < 20:
        errors.append(f"Description too short ({len(desc)} chars): write at least 20 characters")
    else:
        print(f"  PASS  description is present and descriptive")

    print()
    if errors:
        print(f"VALIDATION FAILED: {len(errors)} error(s) found:")
        for e in errors:
            print(f"  FAIL  {e}")
    else:
        print("VALIDATION PASSED: manifest is ready to submit.")
    return len(errors) == 0

validate("tool_manifest.json")

Chapter 4: APIs: The Front Door

The Analogy

You want to order a pizza. You do not walk into the kitchen and instruct the chef. You do not open the refrigerator yourself and assemble the ingredients. You call a phone number or tap a button in an app.

That phone number, that single point of contact, is the API.

An API (Application Programming Interface) is the front door of a server. It is the only place where the outside world is permitted to make contact. Everything behind the door, the database, the business logic, the internal processing, remains hidden. Callers interact with the front door only.

The front door operates by specific rules: you must ring the bell in the correct way, the door opens only for recognised requests, and it responds in a predictable manner. This predictability is what makes APIs powerful. The server decides exactly what the outside world can request and exactly how it will respond.

The Concept

An API is a defined set of endpoints, specific addresses you can send requests to, along with the rules governing those requests.

Think of endpoints as rooms in a building:

  • Room /tools: "Show me all registered tools."
  • Room /tools/file-reader: "Show me details about the File Reader tool."
  • Room /publish: "I want to register a new tool."

Each room has its own access policy. Some rooms are open to anyone. Others require identification. Some rooms allow you to look but not modify anything. Others allow you to leave data behind.

"""
home_api.py  -  A Home Lighting API with four endpoints.
Demonstrates how an API controls access to the system behind it.
"""
from http.server import HTTPServer, BaseHTTPRequestHandler
import json
import urllib.parse

# This is the "database": the state of every light in the house
LIGHTS = {
    "living-room": {"status": "off", "brightness": 0},
    "bedroom":     {"status": "off", "brightness": 0},
    "kitchen":     {"status": "on",  "brightness": 80},
}

class HomeAPIHandler(BaseHTTPRequestHandler):
    def log_message(self, format, *args):
        print(f"  [API] {format % args}")

    def send_json(self, status, data):
        body = json.dumps(data, indent=2).encode()
        self.send_response(status)
        self.send_header("Content-Type", "application/json")
        self.send_header("Content-Length", len(body))
        self.end_headers()
        self.wfile.write(body)

    def do_GET(self):
        path = urllib.parse.urlparse(self.path).path

        # Endpoint 1: GET /lights (list all lights)
        if path == "/lights":
            self.send_json(200, {"lights": LIGHTS})

        # Endpoint 2: GET /lights/:room (get one light)
        elif path.startswith("/lights/"):
            room = path.split("/")[-1]
            if room in LIGHTS:
                self.send_json(200, {room: LIGHTS[room]})
            else:
                self.send_json(404, {"error": f"No room named '{room}'"})

        else:
            self.send_json(404, {"error": f"No endpoint at '{path}'"})

    def do_POST(self):
        path = urllib.parse.urlparse(self.path).path

        # Endpoint 3: POST /lights/:room/on (turn a light on)
        if path.startswith("/lights/") and path.endswith("/on"):
            room = path.split("/")[2]
            if room in LIGHTS:
                LIGHTS[room]["status"] = "on"
                LIGHTS[room]["brightness"] = 100
                self.send_json(200, {
                    "message": f"{room} light is now on",
                    "light": LIGHTS[room]
                })
            else:
                self.send_json(404, {"error": f"No room named '{room}'"})

        # Endpoint 4: POST /lights/:room/off (turn a light off)
        elif path.startswith("/lights/") and path.endswith("/off"):
            room = path.split("/")[2]
            if room in LIGHTS:
                LIGHTS[room]["status"] = "off"
                LIGHTS[room]["brightness"] = 0
                self.send_json(200, {
                    "message": f"{room} light is now off",
                    "light": LIGHTS[room]
                })
            else:
                self.send_json(404, {"error": f"No room named '{room}'"})

        else:
            self.send_json(404, {"error": f"No endpoint at '{path}'"})

Chapter 5: What Is a Registry?

The Analogy

In the early days of smartphone apps, before app stores existed, installing software on a phone was chaotic. You had to find the developer's website, download a file of uncertain origin, hope it was safe, figure out how to install it, and then discover, after all that effort, that it did not work on your device.

Then centralised app stores appeared. Suddenly, one place held everything: every available application, reviews indicating quality, a safety review process, one-tap installation, and automatic updates.

An app store is a registry: a centralised, organised, trusted catalogue of things. The Tool Registry is the same concept applied to AI tools.

The Concept

A registry is an organised, searchable catalogue of items that others can discover and use. You already interact with several registries without thinking of them as such:

App Store / Google PlayMobile applicationsPhone users
npm (npmjs.com)JavaScript packagesWeb developers

The Tool Registry was created to solve a specific problem: AI agents had no reliable, standardised way to discover what tools were available to them. Without a registry, an agent must be manually configured to know about each tool individually. This does not scale. As the number of AI tools grows into the thousands, a searchable directory becomes essential.

What a Registry Entry Contains

Every entry in the Tool Registry represents one AI tool. The following shows the structure of a registry record, based on the real model in the codebase:

// tool-registry/pkg/model/types.go
type Tool struct {
    ID          string      // Unique identifier, e.g. "file-reader"
    Name        string      // Human-readable name
    Description string      // What this tool does
    Publisher   Publisher   // Who created and is responsible for it
    Repository  *Repository // Where the source code lives
    Packages    []Package   // How to install it (npm, pip, docker, etc.)
    Version     string      // Current version
    CreatedAt   time.Time   // When it was first published
    UpdatedAt   time.Time   // When it was last updated
}

Each field answers a question a developer or AI agent would ask:

  • ID: What is the unique identifier I can reference this tool by?
  • Description: What does this tool actually do?
  • Publisher: Who is responsible for it, and can I trust them?
  • Packages: How do I install and run it?
  • Version: Which version am I getting?

The registry exists to create, store, and serve these records reliably.

"""
query_registry.py
Demonstrates the three core operations every registry must support:
  1. List all entries
  2. Get one entry by ID
  3. Search entries by keyword
"""
import sqlite3
import json

DB = "tool_registry.db"

def list_tools(conn, limit=10):
    """Return all tools, sorted by name."""
    rows = conn.execute(
        "SELECT id, name, description, version, publisher, category "
        "FROM tools ORDER BY name LIMIT ?",
        (limit,)
    ).fetchall()
    return [
        {"id": r[0], "name": r[1], "description": r[2],
         "version": r[3], "publisher": r[4], "category": r[5]}
        for r in rows
    ]

def get_tool(conn, tool_id):
    """Return one specific tool by its ID."""
    row = conn.execute(
        "SELECT id, name, description, version, publisher, category "
        "FROM tools WHERE id = ?",
        (tool_id,)
    ).fetchone()
    if row:
        return {"id": row[0], "name": row[1], "description": row[2],
                "version": row[3], "publisher": row[4], "category": row[5]}
    return None

def search_tools(conn, keyword):
    """Return tools whose name, description, or category matches the keyword."""
    rows = conn.execute(
        "SELECT id, name, description, version FROM tools "
        "WHERE name LIKE ? OR description LIKE ? OR category LIKE ?",
        (f"%{keyword}%", f"%{keyword}%", f"%{keyword}%")
    ).fetchall()
    return [{"id": r[0], "name": r[1], "description": r[2], "version": r[3]}
            for r in rows]

def count_tools(conn):
    return conn.execute("SELECT COUNT(*) FROM tools").fetchone()[0]

🌍 Real World

npm, the Node Package Manager registry, hosts over two million JavaScript packages and processes more than three billion downloads per day. When a developer runs npm install express, npm's registry receives a query, finds the package entry, and returns the download location. The entire operation follows the same three-step pattern you just built: receive the request, query the database, return the result. The Tool Registry follows the same model, with AI agents substituted for human developers as the primary consumers of the catalogue.

The Takeaway

A registry is an organised, searchable catalogue. Its three core operations are universal: list everything, retrieve one specific item, and search by keyword. The Tool Registry applies this pattern to AI tools, giving agents a reliable way to discover what capabilities are available to them.
